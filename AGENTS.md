# AGENTS.md

## 项目角色

本仓库是一个 OKX 现货量化交易系统训练场。第一期目标不是追求收益，而是在模拟盘和小资金实盘中验证完整交易链路：行情、成本、策略意图、风控、OMS、成交、账本、对账和告警。

## 硬性纪律

1. 任何涉及交易所接口、字段含义、费率、精度、订单状态、WebSocket 行为、模拟盘行为的开发，必须先查官方文档。
2. 官方文档不足时，可以参考成熟开源项目或 SDK，但必须在 `docs/EVIDENCE.md` 记录来源、结论和影响到的代码路径。
3. 不允许凭记忆编造 API 参数、字段名、状态值、错误码、费率规则或交易限制。
4. 策略层不得直接下单。策略只能输出 `TradeIntent` 或目标仓位，必须经过风控和 OMS。
5. 实盘 API Key 只能使用 `Read + Trade` 权限，严禁开启 `Withdraw`。
6. `demo` 和 `live` 配置必须分离，实盘开关必须显式配置。
7. 对账失败、订单状态未知、行情过旧、API 错误过多时，系统必须停止创建新订单。
8. 所有资金相关计算必须使用 `Decimal`，禁止用浮点数处理价格、数量、手续费和盈亏。
9. 每个核心模块至少有单元测试覆盖状态转换、精度处理、风控拒单这类容易造成资金错误的逻辑。
10. 不要修改或删除旧的研究数据文件，除非用户明确要求。

## 依据优先级

1. OKX 官方 API 文档、OKX 帮助中心、TradingView 官方文档。
2. 成熟开源项目的源码或文档，例如 Freqtrade、Hummingbot、NautilusTrader、Jesse。
3. 其他博客、视频、论坛只能作为线索，不能作为实现依据。

## 开发流程

新增外部接口能力时：

1. 在 `docs/EVIDENCE.md` 追加依据。
2. 在相关代码注释或文档中标明依赖的官方概念。
3. 增加最小测试或手工验收步骤。
4. 更新 `ROADMAP.md` 的进度。

修改交易状态机、风控、对账逻辑时：

1. 先写清楚状态或规则。
2. 补测试。
3. 再改实现。
4. 运行测试后再继续下一步。

## API Key 处理

不要把 API Key、Secret、Passphrase 写进仓库。使用 `.env.demo` 和 `.env.live` 这类本地文件，并确保它们被 `.gitignore` 忽略。聊天中也不需要发送真实密钥。
